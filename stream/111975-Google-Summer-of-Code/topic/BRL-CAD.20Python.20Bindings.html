Sean (Feb 23 2026 at 19:17):

It will help for you to be at least minimally familiar with the existing libraries involved, which will likely be LIBRT (main geometry library, huge), LIBWDB (library for writing/creating geometry, small), and the underlying helper libraries it exposes (LIBBU for basic utilities, LIBBN for basic numerics, LIBBG for some basic geometry helper routines). Be sure to look at the prior efforts by kanzure and nmz787 to see how they approached it!

At a much higher level just for awareness is LIBGED which is a command library that has all 350 or so commands exposed to users (it's a scripting argc/argv-style string API). There's also a prior effort to expose Python as a 1st-class procedural primitive, so you could create and raytrace geometry dynamically, on the fly, by creating a new entity type that runs code that generates and evaluates geometry on the fly. Involving either would be a project in themselves, but helps to understand the contexts and prior work.

Daniel Rossberg (May 16 2026 at 14:41):

Some remarks:

The C keywords (function and type names, ...) start with Brl.
Because C doesn't know about namespaces, it's usual to use prefixes to distinguish features from different libraries or modules. BRL-CAD core libraries use the brl_ prefix to mark BRL-CAD versions of some functions. For the C++ classes, I use BRLCAD as namespace. But, Brlcad or BrlCad as prefix for C would be a bit to long. So, I ended up with Brl. In addition, I want to stay with the CamelCase naming convention in MOOSE.

The keywords for the C API are directly derived from the corresponding C++ names:

  • BrlConstDatabase => BRLCAD::ConstDatabase
  • BrlConstDatabaseLoad => BRLCAD::ConstDatabase::Load

I created a Handle class.
The motivation is to have more save casts. Every handle for a C++ class has the same type in C. There were given different names, e.g. BrlConstDatabase, but they are all void* at the end. If nobody makes a mistake, things like the old Constdatabase.cpp will work. But, something really bad can happen, if the handles will be mixed up.

That's why I put a magic number at the beginning of the C++ classes, very much like in the BRL-CAD core's structures. An alternative would have been to use a proxy structure like

struct {
    const char* magic;
    void*       instance;
};

But, this makes only sense if

  • The proxy would be used on the client's side as automatic (i.e. on the stack) variable or
  • It wouldn't be possible to change the C++ classes.

However, in our case we had to manage two pointers instead of one with this solution. That's why I rejected it.

The magic numbers in Handle are memory addresses: pointers to strings. To check them, only the pointers have to be compared. The strings behind them are useful for debugging.

Alternatively to my solution, the list of magic addresses could be static const members of the Handle class. But, I want to express that they are somehow private to the classes they belong, and only published them in Handle.h.

BTW, the destructor of Handle has to be virtual, at least if build with the GNU compiler. That's because the derived classes have virtual functions and the first entry in their instances binary representations is the pointer to their virtual functions table. If Handle would not be a virtual class itself, after a hard cast to Handle, m_magic would point to the virtual functions table, not the magic number.
